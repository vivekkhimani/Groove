    Groove: An Unsupervised, Data Sensitive Music Recommendation Algorithm

    An unsupervised approach to music recommendation without accessing other users data or using traditional recommendation algorithms like collaborative filtering.

    About Us

    Spotify and other popular music apps rely heavily on other users data to tailor the best recommendations from millions of options available. Traditionally, they find the music in your playlist and find other users with similar interests. After matching two or more users with similar interests, the algorithm recommends the music that a user might not have tried before, but is liked by another user with similar taste. Of course, the companies don't disclose the exact details about their algorithms as it helps them stand out from the competitors. However, the approach described above is usually a backbone of any music recommendation algorithm.

    However, with stricter legislations (like GDPR) and increased awareness about privacy, it might not be always possible to access, store, and use other user's data. As the preferred music and playlists can be strong indicator about the psychological conditions of a listener, music might soon get classified as a sensitive user data. Additionally, if two or more users have same taste for a specific song, it cannot be implied that they might share the taste for other songs. At times, a user might only like a specific song due to favorite artist, popular album, or even the actors. In that case, a liked song might introduce a bias and might degrade the performance of the recommender. Lastly, we believe that users often tend to like songs with similar intrinsic sound wave characteristics like beats, tempo, timbre, pitch, frequency, etc.

    As a result, we introduce Groove: An Unsupervised, Data Sensitive Music Recommendation Algorithm.Groove aims to address the issues mentioned in the previous paragraphs and is built using powerful, unsupervised clustering algorithm that is trained on vectorized sound wave characteristics data. By introducing an unsupervised recommendations approach instead of the traditional approaches, we provide highly personalized, private, and bias-free recommendations. More importantly, our system is implemented to collect and improve based on the recommendation feedback collected from the users.

    FAQ

    Why are the users required to log in using Spotify?

    We claim to not access other user's data, which is normally used by the traditional recommendation algorithms. Traditional algorithms use collaborative filtering, which heavily relies on other user's data. We ask users to log in using Spotify so that we can compare the wave characteristics of their preferred Spotify songs and make unsupervised recommendations from our library. Users can choose to get recommendations based on "Top 20 Spotify Songs", "Saved Songs", "Recently Played Songs in 24 hours", or any of the playlists. This concept even works without getting preferences through Spotify. In that case, users would have to manually enter their preferred songs which is highly inconvenient.

    How do we make recommendations without using other user's data?

    Traditional algorithms use a popular collaborative filtering algorithm which finds other users with similar music taste as yours and make recommendations from their library which you might not have heard before. In our case, we have compiled a huge library of Spotify songs and trained an unsupervised clustering algorithm based on implicit sound wave characteristics. Once we get your preferences from Spotify, our algorithm will select an appropriate cluster for your song based on the wave characteristics. We will pick the closest song from the determined cluster and recommend it to you. As a result, we will only use your preferences and clusters generated from our library to make recommendations. We will not use other user's data.

    How is the algorithm reliable?

    We believe that the users usually love songs with similar sound wave characteristics like pitch, tempo, beats, frequency, timbre, acousticness, speechiness, etc.. While traditional algorithms assume that two or more users with similar taste on a few songs will like all the songs in common, we have a different perspective. Sometimes it's possible that if two or more users have same taste for a specific song, it's not necessary that they might have same taste for other songs. At times, the user might only like a specific song due to biases like favorite artists, albums, or even the actors. Instead, we believe that the users will like the songs which have similar implicit sound wave characteristics to their previously liked songs. Peek into your playlist and think if most of your songs have similar tempo, beats, frequency, genre, etc. As we recommend songs based on such characteristics, our algorithm is extremely reliable.

    How do existing recommendation algorithms work?

    The existing algorithms heavily rely on collaborative filtering which makes recommendations by matching two or more users with similar taste. More details about the traditional music recommendation algorithms can be found here.

    How is our algorithm different from existing algorithms?

    The traditional algorithms heavily rely on other user's data and it's dangerous to assume that we would always have access to other user's data. Also, due to stricter data privacy legislations, accessing user data for Internet-based services is becoming more and more difficult. Primarily motivated by this issue, our algorithm strictly uses the implicit sound wave characteristics of your previously liked or preferred songs and make ML-based recommendations from our libary.

    What is the primary motivation for developing such an algorithm?

    • Stricter privacy-related legislations (GDPR) make it harder to access other users data and apply the traditional recommendation algorithms like collaborative filtering
    • As traditional algorithms are likely to recommend songs which are highly popular with several users, the new artists and songs don't get enough chances to stand out. As we don't treat popularity as a ML feature, our algorithm treats all the artists and songs equally.
    • Making recommendations based on implicit sound wave characteristics (pitch, tempo, beats, timbre, frequency, etc.) provides greater personalization to the users and makes the recommender free from uncontrolled biases like favorite artists, favorite album, etc.

    How big is the library used for making recommendations?

    If we were to use this algorithm in production, we could have trained it on an extremely huge library of songs. For example, Spotify has ~50 million songs. However, as we didn't have enough resources to handle such a huge amount of data, we trained a demo algorithm using 170,000 Spotify songs which are randomly compiled from 1938-2020. Because of random compilation, you might see some irrelevant recommendations. However, we believe that the algorithm could do better if trained on larger data.

    What ML algorithms have been used to make recommendations?

    As we didn't have access to labelled data from other users, we have used unsupervised machine learning algorithm. We primarily used K-Means Clustering & mini-batch K-Means Clustering to classify 170,000 songs in our library into 1000 clusters. As each of our songs had 9 features (sound wave characteristics), which was a lot for effective K-Means clustering, we used Principal Component Analysis (PCA) for dimensionality reduction while preserving the sound wave characteristics. After evaluating the model performance using multiple hyperparameters (PCA-dimensions, number of clusters, etc.), we settled with 2 features and 1000 clusters. We used metrics like Sum of Squared Error (SSE) Score and Silhouette Score to determine the appropriate hyperparameters. In the current demo, we are using a K-Means Clustering algorithm trained for 1000 clusters. Each song has 2 features after the application of PCA.

    What's the exact process of making recommendations?

    • We trained a K-Means clustering algorithm using our dataset that contained 170,000 songs, and 9 features for each song.
    • Our algorithm was tuned to classify the songs in 1000 clusters based on the features (sound wave characteristics).
    • Users are required to log in through Spotify which will take them to our homepage.
    • Users are provided with an option to select the criteria for getting recommendations. They can select to get the recommendations based on one of the following:
      • Top 20 Song
      • Recently Played Songs (24 hours)
      • Saved Tracks
      • Playlists created by the user
    • Once the user selects any of the aforementioned criteria, we will use the Spotify API to extract their preferred songs, vectorize it, and feed it to our trained model.
    • Based on the song characteristics, our model will assign a one of the thousand clusters (0 - 999) to each of the song.
    • We will use our library to search other songs that belong to the same cluster as the user preferred song and recommend the closest one.
    • We will recommend the top 10 songs using the same method. User will have an option to approve or disapprove of our recommendations and the system will automatically update the algorithm accordingly.

    How do we personalize the recommendations?

    • As our algorithm makes recommendations based on implicit sound wave characteristics of the previously liked songs by the users, it will be relevant to the taste of the users.
    • As the user has an option to approve or disapprove of our recommendations, we improve the algorithm and make sure that the same recommendation will not be made to the users.
    • Once we will collect enough approval/disapproval data, we will retrain our model using the user preferences, which will act as a labelled data for us.

    What is the future scope of improvement?

    This project can be expanded upon and the results can be drastically improved using one of the following methods:
    • Currently we only have 170,000 songs in our library. We can further improve the results if the model is trained with more songs.
    • Currently, no filters are applied to the recommendations made by ML algorithm. We can apply filters like year of release, language, genre, etc. to make the recommendations more relevant.
    • We are already collecting user approval data. Once we have enough data, we can retrain the ML algorithm with labelled data and a supervised approach to further improve the results.
    • K-Means Clustering algorithm is just a starting point for this problem. We can explore more algorithms (probably time-series based clustering) and hyperparameters to get even better results.
    • We can include more features (sound wave characteristics) and perform time-series based clustering to improve the results. This is a great direction for further exploration.

    What could have we done if we had an access to GPT-3?

    GPT-3 is a latest transformer-based model trained by OpenAI on a massive amount of data. GPT-3 can be trained to perform a wide-range of tasks by just training it using a few examples (few-shot learning). Which means, the GPT-3 API can be readily used to give a few training examples and results. The people who have an access to this API have done really cool projects like text to code, text to LaTex, etc. by training the GPT-3 model using a few manually generated examples.

    Unfortunately, as the API is in early stages of development, only a few people have an access to this API and can use it for their own purpose. If we had an access to this API, we could have used it for making recommendations instead of K-Means Clustering. But that would have definitely made our project a lot less challenging because we would have only configured the API instead of manually compiling the data, engineering the features, trying out different hyperparameters, training a model, and deploying it in our application.
